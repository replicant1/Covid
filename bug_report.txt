CRITICAL BUGS
=============

1. [FIXED - PR #10] Missing @Singleton on AppDatabase provider
   File: app/src/main/java/com/rodbailey/covid/core/di/DatabaseModule.kt
   Lines: 18-25
   Detail: The provideAppDatabase() function lacks @Singleton. Despite being installed in
   SingletonComponent, every injection creates a new Room database instance. This causes
   data inconsistency, memory leaks, and unpredictable test failures.
   Fix: Add @Singleton to provideAppDatabase().

2. [FIXED - PR #11] onEach used for long-running suspend side effects
   File: app/src/main/java/com/rodbailey/covid/data/repo/DefaultCovidRepository.kt
   Lines: 46-60
   Detail: getRegionsStream() calls refreshRegions() (a suspend function) inside onEach for
   each emission. While onEach does accept a suspend lambda, this causes long-running work
   to accumulate and previously started refreshes to continue even when new data arrives.
   Fix: Replaced onEach with transformLatest so that in-flight refreshRegions() calls are
   cancelled when a new emission arrives, avoiding overlapping work and matching the
   intended behaviour.

3. [FIXED - PR #26] Missing @Singleton on DAO providers
   File: app/src/main/java/com/rodbailey/covid/core/di/DatabaseModule.kt
   Lines: 27-35
   Detail: provideRegionDao() and provideRegStatsDao() both lack @Singleton. This compounds
   bug #1 — DAOs may be derived from different AppDatabase instances.
   Fix: Add @Singleton to both DAO provider functions.


HIGH PRIORITY BUGS
==================

4. [FIXED - PR #27] HTTP logging interceptor always set to BODY level in production
   File: app/src/main/java/com/rodbailey/covid/data/net/CovidAPIClient.kt
   Lines: 11-13
   Detail: HttpLoggingInterceptor.Level.BODY is set unconditionally, logging full
   request/response bodies in production builds. This is a performance and data
   exposure risk.
   Fix: Gate on BuildConfig.DEBUG:
       level = if (BuildConfig.DEBUG) HttpLoggingInterceptor.Level.BODY
               else HttpLoggingInterceptor.Level.NONE

5. [FIXED] collectLatest on a cold flow triggers multiple concurrent network requests
   File: app/src/main/java/com/rodbailey/covid/presentation/MainViewModel.kt
   Lines: ~150-172
   Detail: loadReportDataForRegion() uses collectLatest on a cold flow returned by
   getRegionStatsStream(). Rapid region taps cancel the previous collection but still
   fire network requests before cancellation completes, creating unnecessary traffic.
   Fix: Cancel the previous job explicitly before launching a new one, or use a
   MutableStateFlow with flatMapLatest to serialize requests.


MEDIUM PRIORITY BUGS
====================

6. Empty stats result produces a misleading generic error
   File: app/src/main/java/com/rodbailey/covid/presentation/MainViewModel.kt
   Lines: ~157-165
   Detail: The original isNotEmpty() guard that caused infinite loading is gone.
   However, if the API returns an empty list, regionStatsList.first() throws
   NoSuchElementException, which is caught by the generic catch block. The data
   panel closes and a generic "failed to load data" error is shown, with no
   indication that the response was simply empty rather than a network failure.
   Fix: Check regionStatsList.isEmpty() before calling .first() and handle the
   empty case explicitly (e.g. show a "no data available" message).

7. Error state never rendered in region list
   File: app/src/main/java/com/rodbailey/covid/presentation/main/MainScreen.kt
   Lines: ~109-125
   Detail: The LazyColumn only handles UIState.matchingRegions is Success. If the state
   is Error, nothing is displayed and the failure is silently swallowed.
   Fix: Add an else/is Error branch to show an error message to the user.

8. Channel with default RENDEZVOUS capacity may drop errors
   File: app/src/main/java/com/rodbailey/covid/presentation/MainViewModel.kt
   Lines: 83-84
   Detail: errorChannel is created with default capacity (RENDEZVOUS = 0). If an error
   is emitted before the UI has subscribed to errorFlow (race condition during
   screen initialisation), the send suspends and the error may be lost.
   Fix: Use Channel(Channel.BUFFERED) to allow at least one error to be queued.
